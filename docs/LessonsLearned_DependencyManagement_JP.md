# 依存関係管理とライブラリ移行に関する教訓 (2026-03-05)

## 1. 名前空間の偽装（Duck Typing的属性）のリスク
- **事象**: `MicroResolver` の `InjectAttribute` を、NuGet 参照を避けるために `namespace MicroResolver` でローカル定義していた。
- **問題点**: 
    - フォルダ構造と名前空間が一致せず、静的解析やリファクタリング時に混乱を招く。
    - 「依存していない」と言いつつ、コード上は特定ライブラリの名前空間を前提としており、不透明な結合が生じていた。
- **教訓**: コアロジックの純粋性を保つ意図は重要だが、名前空間を偽装するメリットよりも、構成の不透明さによるメンテナンスコストの方が大きくなる場合がある。現代的な .NET 開発では、軽量なライブラリであれば直接 NuGet 参照し、透明性を高める方が望ましい。

## 2. メジャーバージョンアップ（Tomlyn 0.20 -> 1.0.0）への対応
- **事象**: `Tomlyn` が 1.0.0 になり、TOML 1.1 仕様への準拠と API の大幅刷新が行われた。
- **変更内容**:
    - `Toml.ToModel` / `Toml.FromModel` -> `TomlSerializer.Deserialize` / `TomlSerializer.Serialize`
    - `TomlModelOptions` -> `TomlSerializerOptions`
    - 命名ポリシーの指定方法が `System.Text.Json.JsonNamingPolicy` 準拠に変更。
- **教訓**:
    - OSS ライブラリのメジャーアップデートでは、API デザインが標準的な .NET イディオム（この場合は `System.Text.Json`）に寄せられる傾向がある。
    - 破壊的変更がある場合、単なる置換だけでなく、周辺の `using` （`System.Text.Json` など）の追加が必要になる。

## 3. 依存関係の競合 (NU1605) とバージョン統一
- **事象**: 一部のプロジェクト（Core）だけを先行してアップグレードした結果、他のプロジェクト（UI.Cli など）でパッケージのダウングレード警告が発生し、ビルドが停止した。
- **教訓**: ソリューション内で共有されるライブラリ（特に Core や共通のユーティリティ）のバージョンを変更する際は、依存するすべてのプロジェクトでバージョンを統一する必要がある。`Directory.Packages.props` (Central Package Management) の導入を検討する動機となる。

## 4. 証跡の重要性
- **事象**: 移行中に多くのビルドエラーやパッケージ競合が発生したが、一時的なテストコードやブラウザでのドキュメント調査を通じて一つずつ解消した。
- **教訓**: 複雑な移行作業では、何を確認し、どの API が正解だったかを随時記録することで、手戻りを防ぎ、最終的な「教訓」として資産化できる。
